## 2024-06-15 - Prevent User Enumeration via Timing Attacks in Login
**Vulnerability:** The `Login` handler returned early immediately if the provided email did not exist in the database (`sql.ErrNoRows`), without computing a bcrypt hash. Because bcrypt hashing is computationally expensive (taking ~2.6s at cost 14), a successful database hit took significantly longer than a miss. Attackers could exploit this timing difference to enumerate valid user emails.
**Learning:** Early returns in authentication flows, while generally good for performance, can introduce critical side-channel vulnerabilities. The time taken to process an authentication request must be constant (or functionally indistinguishable) regardless of whether the username is valid or invalid.
**Prevention:** Always execute the computationally expensive operation (e.g., password hashing/comparison) against a constant "dummy hash" when a user is not found. This ensures the CPU time spent processing the request remains consistent.
